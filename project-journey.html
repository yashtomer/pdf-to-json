1. Project overview

The goal: read four real PDF files (two digitally-signed NICSI Work Orders + two scanned Monthly Performance Reports) and produce clean, structured JSON for every document — handling both native-text PDFs and image-only scans without errors.

The challenge: the four PDFs span three very different formats:

These are native programs — not Python packages. Install them once at the OS level. The Python libraries are thin wrappers that shell out to these.

4. Python libraries

4.1 Direct dependencies — the 3 we chose

These are the only ones declared in pyproject.toml. They're what the code actually imports.

LibraryRole in this projectLicense
pdfplumber Primary PDF reader. Extracts both text and table cells from native-text PDFs, with layout awareness. Multi-line wrapped cells come back as one string with internal \n separators. MIT
pdf2image Renders a PDF page into a PIL image. Only needed for scanned PDFs (where there is no text layer). Shells out to poppler. MIT
pytesseract Python wrapper around the Tesseract OCR engine. Takes a PIL image, returns extracted text. The actual recognition happens in tesseract (C++). Apache 2

5. Issues & how we fixed them

Listed in the order we hit them. Each one shows the problem, the root cause (one sentence), and the fix.

1 2 of 4 PDFs returned essentially no text

bugfeature

Problem

The two NICSI Work Orders extracted 17–22 KB of clean text. The two MPR files extracted 1–3 characters total — almost nothing.

Root cause

Those files are scanned images (one from a VersaLink scanner, one from "Microsoft Print To PDF" of an image). They have no text layer for extract_text() to read.

Fix

Built a two-tier strategy: try pdfplumber.extract_text() first; if a page returns fewer than 30 characters, re-render the page as an image and OCR it with Tesseract. The threshold (MIN_CHARS_PER_PAGE) is tunable.

2 No requirements.txt in the project

tooling

Problem

Switched the project to uv for dependency management — but anything that doesn't speak uv (legacy CI, Docker, plain pip users) needs a requirements.txt.

Fix

Exported a hand-curated requirements.txt containing only the 3 direct dependencies, with inline comments explaining what each one is for. uv.lock remains the source of truth for byte-for-byte reproducibility.

# pdfplumber: extracts text + tables from native-text PDFs.
pdfplumber==0.11.9
# pdf2image: renders a PDF page to an image (for OCR).
pdf2image==1.17.0
# pytesseract: Python wrapper around the Tesseract OCR engine.
pytesseract==0.3.13

3 Multi-line table cells broke across rows

bug

Problem

The "Description" cell "Software Application Support Engineer (4 to less than 6 years relevant experience)-3rd year 2nd Increment" came back as four separate output lines because the PDF wraps it visually across four lines inside one cell.

Root cause

The code was using page.extract_text(), which returns text in reading order with line breaks where the layout has line breaks — it doesn't understand table-cell boundaries.

Fix

Switched to page.extract_tables(), which returns each table cell as one string (with \n separators inside it for wrapped content). Then collapsed those internal newlines into single spaces with a helper.

4 Headers came out as gibberish: "No of Person s Require d"

bug

Problem

The "No of Persons Required" column header was extracted as "No of Person s Require d" — the words "Persons" and "Required" were split because the PDF stacks them vertically across narrow columns.

Root cause

Naive newline-to-space replacement turns "Person\ns" into "Person s" (with a space) instead of joining as "Persons".

Fix

Added a heuristic: when a line break leaves a single lowercase letter on its own line, treat it as a word-wrap continuation and join without a space. Catches "Persons" and "Required" while still keeping legitimate two-word breaks like "No of".

5 Numbers were split mid-value: "14,09,808.9 8"

bug

Problem

Total-amount values like "14,09,808.98", "2,53,765.62", and "56,844.80" were coming out with a stray space — same vertical-wrap problem as the headers, but for digits.

Fix

Extended the heuristic with two more rules: a single trailing digit joins to a number ("…9" + "8" → "…98"), and a decimal/comma continuation joins to a number ("765" + ".62" → "765.62").

6 Tables were positional (no column names)

ux

Problem

Each row was a list ["1", "998313", "Software Application…", ...] — you had to know the column index to find any value.

Fix

Built _table_to_keyed(): treats row 0 of every table as the column headers, then turns each subsequent row into a dict keyed by those headers. Every row is now self-describing.

7 Two confusing fields: tables and tables_keyed

ux

Problem

Adding tables_keyed as a separate field next to tables doubled the JSON size and made it unclear which one to read.

Fix

Removed the raw positional tables entirely. The keyed list-of-dicts version is now the only tables field. Tables without real headers are routed elsewhere (see issue 10).

8 Keys had spaces, capitals, and special chars

ux

Problem

Keys like "S. No", "HSN/ SAC Code", "CGST (%)" are awkward in Python (can't be used as attributes), JS (need bracket notation), and most query languages.

Fix

All keys are now snake_case: lowercase, % → pct, any run of non-alphanumeric characters → single _, leading/trailing _ stripped. Empty headers → col_1, col_2. Duplicate names get _2 suffixes.

"S. No"                      → "s_no"
"HSN/ SAC Code"              → "hsn_sac_code"
"CGST (%)"                   → "cgst_pct"
"Required Period (Months)"   → "required_period_months"

9 Two values stuffed into one cell: "18.00% 56,844.80"

feature

Problem

The PDF visually stacks the tax rate on top of the tax amount inside one visual cell. The cell header "CGST (%) /Amount" covers both values, and pdfplumber correctly returns them as one cell — but downstream code wants them separate.

Fix

Added an opt-in --split-tax-cells flag. When set, any header matching <name>_pct_amount is split into two keys (<name>_pct and <name>_amount), and the cell value is split at the first %. Totals rows (with only the amount) get an empty rate field — no fabrication.

10 Header info was buried in text, not structured

feature

Problem

Things like Work Order No, Date, Name, Phone, Email sit in the document-header box at the top — they were captured as one giant text string but not as structured fields.

Fix

Added a fields dict per page. Detection: any "table" where ≥20% of cells end with : / :- (or match well-known bare labels like Date) is treated as a key-value box, flattened into fields, and omitted from tables. So tables contains only real data tables, and fields contains structured metadata.

11 Column-letter "(A) (B) (C)…" row appeared as data

bug

Problem

The sub-header row in the line-items table (used by the formula (AxBxC) in the column headers) was being treated as a regular data row.

Fix

Added _is_column_letter_row(): if a row's non-empty values are mostly single uppercase letters in parentheses ((A), (B), …), drop the row when building tables.

12 Continuation pages (2–3) had empty fields

feature

Problem

Pages 2 and 3 each have a small header line like "Project No.: S220466MPTS PO No.: M2602757" — useful info, completely missed because pdfplumber doesn't detect a "table" there.

Fix

Added _extract_text_fields() — a small regex pass over the page text that captures common patterns: work_order_no, project_no, po_no, empanelment_no, valid_till, gstin, subject, mpr_for_month, document_type. Table-extracted fields always win for the same key (setdefault).

13 Signature block and Copy To list weren't structured

feature

Problem

The last page has a signature block (Digitally signed by …, Date: …, (Name), Manager & Project Manager) and a numbered Copy To: distribution list. None of it was being captured as fields.

Fix

Wrote dedicated parsers: _extract_signature_block() handles both digitally-signed (5 fields) and printed-only (3 fields) formats — making the Digitally signed by / Date: lines optional in the regex. _extract_copy_to() captures the numbered list as list[str]. The fields type was upgraded to dict[str, str | list[str]] to support mixed values.

14 Scanned PDFs produced fields: {}

bug

Problem

The MPR JSON was effectively empty after we removed the bulky text field — no fields, no tables.

Root cause

The _extract_text_fields() call was inside the if method == "text" branch — it never ran on OCR text.

Fix

Moved the call after both branches so it runs on page_result.text regardless of how it was obtained. Result: the scanned MPR JSON now has 4 structured fields per page (work_order_no, project_no, mpr_for_month, document_type).

15 OCR misread an "S" as "$": "$211164MPMH"

bug

Problem

One scanned MPR had project_no: "$211164MPMH". The leading S was being misrecognized by Tesseract.

Investigation

Tested at multiple DPIs:

DPIResult
300✗ $211164MPMH
400✓ S211164MPMH
500✗ $211164MPMH (counter-intuitively worse — upsampling artifacts)
Fix

Bumped the default OCR DPI from 300 → 400. It's also the value Tesseract's own documentation recommends as the sweet spot for typical scanned documents.

16 The text field made JSON files huge and hard to scan

ux

Problem

JSON files were ~10KB each, mostly the raw text blob. Scrolling past it to find fields / tables was annoying in an editor.

Fix

Excluded text from JSON output by default — files dropped to ~4KB. Added --include-text to opt back in. --dump-text DIR still works for getting the full text as separate .txt files.

17 Sample PDF filenames had spaces / mixed case / commas

tooling

Problem

Filenames like "MPRs for the month of Feb, March, April, 2026.PDF" required quoting in every shell command and broke glob patterns.

Fix

Renamed all sample PDFs to lowercase snake_case with no spaces and no commas. samples/*.pdf globs work cleanly. The folder is also gitignored to keep sensitive content (employee names, salaries) out of version control.

7. How to run it

One-time setup

# macOS
brew install tesseract poppler uv

# Linux (Debian/Ubuntu)
apt install tesseract-ocr poppler-utils
# install uv via the official installer (see docs.astral.sh/uv)

# In the project folder
uv sync

Common commands

# Run against every sample PDF, write structured JSON
uv run pdf-reader samples/*.pdf --split-tax-cells --dump-json out/

# Single file, also dump raw text alongside
uv run pdf-reader samples/m2602757signed.pdf --dump-text out/ --dump-json out/

# Include the raw 'text' field in JSON (excluded by default)
uv run pdf-reader samples/*.pdf --include-text --dump-json out/

# Crank OCR resolution higher for difficult scans
uv run pdf-reader scan.pdf --dpi 600 --dump-json out/

Inspecting the output

# Pretty-print one file
jq . out/m2602757signed.json

# Just the fields per page
jq '.pages[] | select(.fields | length > 0) | {page: .page_number, fields}' out/m2602757signed.json

# Just the line items
jq '.pages[].tables[] | .[]' out/m2602757signed.json
